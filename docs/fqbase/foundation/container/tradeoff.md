## 概述

服务容器设计过程中做出了多个技术权衡，理解这些权衡有助于更好地使用和扩展容器。

## 权衡 1: 依赖注入 vs 服务定位器

### 考量因素

| 因素 | 依赖注入 | 服务定位器 |
|------|---------|-----------|
| 耦合度 | 更低 | 中等 |
| 可测试性 | 更容易 | 需要 mock |
| 代码复杂度 | 较高 | 较低 |
| 学习曲线 | 较陡 | 较平缓 |

### 决策

**选择了同时支持两种模式**：
- 优先推荐依赖注入（通过构造函数）
- ServiceLocator 作为补充（用于框架集成）

### 后果

**正面：**
- 灵活性：用户可根据场景选择
- 兼容性：与现有代码共存

**负面：**
- 需要理解两种模式
- 可能导致滥用 ServiceLocator

---

## 权衡 2: 运行时 vs 编译时注册

### 考量因素

| 因素 | 运行时注册 | 编译时注册 |
|------|-----------|-----------|
| 灵活性 | 高 | 低 |
| 性能 | 稍低 | 更高 |
| 类型安全 | 较低 | 较高 |
| 复杂度 | 较低 | 较高 |

### 决策

**选择运行时注册**

```python
container.register_singleton(Interface, Implementation)
```

### 后果

**正面：**
- 简单易用
- 配置灵活
- 支持热更新

**负面：**
- 运行时错误
- IDE 支持有限

---

## 权衡 3: 反射 vs 类型提示

### 考量因素

| 因素 | 反射（自动注入） | 手动声明依赖 |
|------|-----------------|-------------|
| 便捷性 | 高 | 低 |
| 明确性 | 低 | 高 |
| 性能 | 稍低 | 更高 |
| 可维护性 | 差 | 好 |

### 决策

**选择混合模式**：
- 支持手动声明依赖 (`dependencies` 参数)
- 自动尝试构造函数注入

```python
# 手动声明
container.register_singleton(
    UserService,
    UserService,
    dependencies=[Database, Cache]
)

# 自动尝试
class UserService:
    def __init__(self, db: Database, cache: Cache):
        # 容器自动尝试注入
```

### 后果

**正面：**
- 灵活：可根据需要选择
- 兼容：支持两种风格

**负面：**
- 规则复杂
- 调试困难

## 权衡 4: 生命周期选择

### 考量因素

| 因素 | SINGLETON | TRANSIENT | SCOPED |
|------|-----------|-----------|--------|
| 性能 | 最高 | 较低 | 中等 |
| 内存 | 最低 | 较高 | 中等 |
| 隔离性 | 低 | 高 | 中等 |
| 实现复杂度 | 中等 | 低 | 较高 |

### 决策

**支持所有三种生命周期**

### 后果

**正面：**
- 满足各种场景
- 性能可优化

**负面：**
- 需要理解差异
- 选择困难

---

## 权衡 5: 循环依赖检测策略

### 考量因素

| 因素 | 构建时检测 | 运行时检测 |
|------|-----------|-----------|
| 及时性 | 早 | 晚 |
| 性能开销 | 较高 | 较低 |
| 复杂度 | 高 | 低 |

### 决策

**选择运行时检测**

```python
# 在 get() 过程中检测
_resolving: Set[Type]  # 维护正在解析的服务

def get(self, service_type):
    if service_type in self._resolving:
        raise CircularDependencyException(...)
```

### 后果

**正面：**
- 实现简单
- 支持动态注册

**负面：**
- 运行时才报错
- 需要测试覆盖
