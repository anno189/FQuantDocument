## 概述

本指南帮助架构师判断何时使用事件总线以及如何选择合适的配置。

## 决策树

```
需要使用事件总线吗？
    │
    ├── 模块需要通信吗？
    │       │
    │       ├── 是 ──▶ 模块是独立开发的吗？
    │       │           │
    │       │           ├── 是 ──▶ 事件总线是好的选择
    │       │           │
    │       │           └── 否 ──▶ 直接调用更简单
    │       │
    │       └── 否 ──▶ 不需要事件总线
    │
    └── 需要解耦吗？
            │
            ├── 是 ──▶ 事件总线是好的选择
            │
            └── 否 ──▶ 直接调用更简单
```

## 场景 1: 微服务内部通信

### 问题

微服务内部各组件需要通信

### 选项

| 选项 | 优点 | 缺点 |
|------|------|------|
| 事件总线 | 解耦、灵活 | 性能开销 |
| 直接调用 | 简单、高效 | 强耦合 |
| 消息队列 | 跨进程 | 复杂性高 |

### 决策

**事件总线** - 适用于同一进程内的组件通信

## 场景 2: 需要事件历史

### 问题

需要记录和回溯事件

### 选项

| 选项 | 优点 | 缺点 |
|------|------|------|
| 事件总线 | 内置历史 | 内存有限 |
| 日志系统 | 持久化 | 需要额外配置 |

### 决策

调试用事件总线，生产用日志系统

## 反模式警示

### 错误示例

```python
# 过度使用：简单调用也用事件总线
def add(a, b):
    return a + b

bus.subscribe('add', add)
bus.publish(Event('add', {'a': 1, 'b': 2}))
```

### 正确做法

```python
# 直接函数调用
result = add(1, 2)
```

## 决策检查清单

在决定使用事件总线前，请检查：

- [ ] 模块是否需要解耦？
- [ ] 是否需要一对多通知？
- [ ] 是否需要动态订阅关系？
- [ ] 事件处理是否异步？

如果以上大部分是"是"，事件总线是一个好选择。
