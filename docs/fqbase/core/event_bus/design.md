## 设计原则

| 原则 | 应用 |
|------|------|
| 单一职责 | EventBus 只负责发布订阅，EventHistory 负责历史，Subscription 负责订阅者信息 |
| 开闭原则 | 通过订阅机制扩展，无需修改核心代码 |
| 依赖倒置 | 依赖抽象回调接口，不依赖具体实现 |
| 最小惊讶 | API 设计简洁直观，符合 Python 惯例 |

## 设计决策

### 决策 1: 单例模式

**上下文：** 需要确保全局只有一个事件总线实例

**决策：** 使用单例模式确保全局唯一

**后果：**
- 正面：所有模块使用同一个事件总线，事件互通
- 正面：简化实例管理
- 负面：测试时需要重置单例（提供方法）

### 决策 2: 弱引用默认不开启

**上下文：** 订阅可能持有对象引用导致内存泄漏

**决策：** 弱引用默认关闭，需要显式开启

**后果：**
- 正面：简单场景无需关心弱引用
- 正面：性能略好（无需解析弱引用）
- 负面：需要开发者主动管理内存

### 决策 3: 异常静默处理

**上下文：** 事件处理中的异常不应中断其他订阅者

**决策：** 捕获并记录异常，不传播

**后果：**
- 正面：一个订阅者错误不影响其他订阅者
- 负面：可能隐藏错误，需要查看日志

### 决策 4: 同步默认

**上下文：** 需要平衡简单性和性能

**决策：** publish() 为默认同步方式

**后果：**
- 正面：简单场景无需额外学习
- 正面：低延迟
- 负面：耗时操作会阻塞

## 使用的模式

| 模式 | 位置 | 用途 |
|------|------|------|
| 单例模式 | EventBus | 确保全局唯一实例 |
| 观察者模式 | subscribe/publish | 解耦发布者和订阅者 |
| 策略模式 | priority | 不同优先级策略 |
| 备忘录模式 | EventHistory | 保存和恢复状态 |

## 扩展点

| 扩展点 | 方法 | 描述 |
|--------|------|------|
| 自定义历史 | 子类化 EventHistory | 自定义历史存储 |
| 自定义过滤 | subscribe | 添加条件过滤 |
| 插件化 | subscribe_global | 全局拦截器 |
